# 房源文字富化價值分析 — CE 看不到哪些特徵、富化哪些需求最值得

**日期**: 2026-06-16
**結論**: CE 訓練/打分用的 `property_to_text` 有兩個盲點(furniture 只取前 5 項、notes 完全不納入),導致使用者**最常問**的需求(採光/安全/安靜/電梯/開伙)CE 全看不到。富化房源文字解鎖這些特徵;C 組實驗已驗證「想要採光好」NDCG 0→1。
**數據來源**: `frontend/assets/property_data.json`(704 房源)
**關聯**: [ce_expansion_augment_experiment.ipynb](../notebooks/ce_expansion_augment_experiment.ipynb)(C 組富化實驗)、[extension_map_training_coverage.md](extension_map_training_coverage.md)、`pipeline/data_prep/precompute_ce_text.py`(Python 端 byte-exact 預算 ce_text)

## 兩個盲點(為什麼 CE 看不到)

`property_to_text`(`pipeline/data_prep/generate_dataset.py`)組房源字串時:
1. **furniture 只取前 5 項** → 平均每房源**漏掉 6.7 個設施**,680/704 房源被砍。連流理台/書桌椅/寬頻網路/飲水機這種常見設施都因排序靠後被丟。
2. **notes/特色欄完全不納入**(僅保留含「寵物/限」的字串)→ 隔音、門禁、採光、補助等軟性特徵全部看不到。

CE(佔最終分 65%)只吃 `property_to_text`,所以這些特徵在 65% 權重的語意層**等於不存在**。

## 富化優先級(常見 × 使用者常問 × CE 現在看不到)

### 🥇 第一優先 — 高覆蓋 + 高頻問 + 完全看不到
1. **採光/通風/不要暗房** → 對外窗(93%)、氣密窗(74%)。*C 組實驗「想要採光好」0→1 的勝利就是這個。*
2. **安全/女生獨居/門禁** → 保全設施(97% 最高覆蓋)、包裹代收(39%)。使用者超常問。
3. **安靜/隔音/怕吵** → 氣密窗(74%)、水泥隔間(79%)。最初「想安靜→隔音房源」的核心訴求。
4. **電梯/不爬樓梯** → 有電梯(84%)。

### 🥈 第二優先 — 中高覆蓋 + 常問
5. **開伙/煮飯** → 可開伙(52%,notes)+ 流理台(26%,furniture 第 6+ 被砍)→ **雙重看不到**。
6. **養寵物** → 可養貓(71%)/可養狗(38%)。
7. **租屋補助/報稅** → 租金補貼(79%)/可報稅(42%)。學生族高頻。
8. **垃圾代收/懶人** → 垃圾代收(78%)。

### 🥉 第三優先 — 低覆蓋但精準
飲水機(12%)、無菸(5%)、獨立電錶(2%)、高齡友善(2%)。

## 實作成果(已落地 2026-06-16)

富化策略簡單:**`property_to_text_enriched` 納入全部 notes + 全部 furniture(不砍前 5)**——`pipeline/data_prep/augment_with_expansion_map.py` 的 `property_to_text_enriched()` 已實作此邏輯,C 組實驗即用此版。C 組富化方案已正式接回 production:

- 前端 `inference.js` `scorePair` 已改餵 `prop.ce_text || prop.text`(讀預算欄,有富化文字就用富化文字)。
- **MAX_LENGTH 已 64 → 128**(`inference-worker.js`;富化文字實測 ~98 token,64 會截斷 99%)。
- 模型已換為 C 組富化 rbt3 student(38.7 MB;舊模型已備份 `.PREV-20260616.onnx`)。
- 訓練/推論皆用同款富化文字,線上線下一致,消除了房源端 enrichment 當初記錄的 OOD 崩壞(見 [ce_text_layer_decision.md](ce_text_layer_decision.md) 後記)。

**推論側刻意不靠前端組裝**:不在前端用 `buildPropText` 重新拼字串(前端基底易與訓練不一致 → OOD)。改用 `pipeline/data_prep/precompute_ce_text.py` 在 Python 端 byte-exact 把富化文字預算進 `property_data.json` 的 `ce_text` 欄,前端只負責讀取。這保證前端餵給 CE 的文字與訓練語料 property 端 byte-for-byte 相同。

## 一句話總結
原 CE 看不到 97% 房源的保全、93% 的採光、79% 的隔音、84% 的電梯——這些正是使用者最常問的。富化房源文字解鎖這些特徵,讓語意層(65% 權重)真正「看見」房源能力,C 組實驗已驗證有效(「想要採光好」0→1、整體 NDCG@5 +0.0125、F1 +0.021)。
