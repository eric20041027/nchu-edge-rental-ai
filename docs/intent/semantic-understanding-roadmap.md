# 邁向「真正的語意理解推薦」— 架構診斷與規劃

> 2026-06-28。在規劃任何架構改動前先做的實證診斷 + 三條路線評估。
> 數據全部本機實跑(現役 974 房源、rbt3 模型、ab_eval_queries fixture),非推測。

## 一、診斷:瓶頸到底在哪(實證)

用現役模型在 6 個隱喻 query 上跑「純語意召回」(cosine top-30,繞過硬約束過濾),
對照 ground truth(ab_eval_queries,GT 已人工核實合理):

| query | #relevant | rbt3 召回@30 | 最佳排名 |
|---|---|---|---|
| 不想追垃圾車 | 16 | 2 | 第 8 |
| 不想出門買水 | 3 | **0** | MISS |
| 怕悶熱 | 9 | **0** | MISS |
| 懶人首選 | 22 | **0** | MISS |
| 淺眠怕吵 | 14 | 5 | 第 2 |
| 要租怕熱 | 3 | **0** | MISS |

- **召回漏掉(relevant 從沒進 top30):4/6**
- **召回到但排序爛(best rank >10):0/6**

→ **瓶頸 100% 在召回的語意理解,精排完全不是問題。**
   CE NDCG@5 ≈ 0.897(修正後實測)佐證精排健康。

## 二、根因(三層,實證揭露)

### 根因 1:精確結構化訊號根本沒進召回向量

GT 房源的共通點是**具體特徵**,而且這些特徵在房源資料裡是**結構化布林欄位**:

| 隱喻 query | 對應房源特徵 | 房源資料裡的欄位 |
|---|---|---|
| 不想出門買水 | 飲水機 | `water_dispenser=true` |
| 不想追垃圾車 | 垃圾代收 | `has_waste_disposal=true` |
| 怕悶熱 / 要租怕熱 | 陽台/對外窗/冷氣 | `has_balcony`、`has_window` |

但 **bi-encoder 召回用的是 `text` 欄位**(精簡關鍵字串:
`住宅 公寓 南區 南和路 15000元 距離2.58km 床 衣櫃 陽台...`),
這些布林欄位(`water_dispenser` 等共 ~15 個)**完全沒餵進召回向量**。
等於要 bi-encoder 從關鍵字串裡「悟」出 query 對應哪個沒寫進去的特徵 —— 結構性不可能。

### 根因 2:換現成「大」encoder 零樣本反而更差(實測,反直覺)

裝 `multilingual-e5-small`(384 維)在同樣 6 query × 974 房源跑對照:

| | rbt3(現役,領域微調過) | e5-small(零樣本) |
|---|---|---|
| recall miss 救回數 | 基準 | **0**(一個都沒救回) |
| 連 rbt3 召回到的「淺眠怕吵」 | 5@2 | 掉到 0 |

→ **「換大 encoder 就好」是錯的直覺。** 零樣本通用 encoder 對「特徵詞袋」格式的房源
   text 反而更弱。大 encoder 要有用,**必須在這個領域 + 這種隱喻↔特徵映射上微調**,
   不是換個更大的權重就行。這也解釋 stage4 為何越破模板越塌縮:rbt3 容量
   塞不下這麼多語意映射,硬塞就犧牲字面區辨。

### 根因 3:stage4 已實證「純資料增強 + rbt3」是死路

ab_eval all R@30:現役 0.26 → v1 破模板 0.073 → v2 合併+全跨域 0.047(越破越退)。
每次都 semantic 微升、keyword 大跌 = 向量塌縮。**rbt3 容量是真瓶頸,但不是
「換大 encoder」單獨能解 —— 要同時解決根因 1(特徵沒進召回)。**

## 三、三條路線評估

### 路線 A 上限量化(實測,78 語意 query)

- **50/78(64%)語意 query** 的 relevant 房源能**完全**靠結構化欄位命中救回。
- **145 個 relevant 房源中 79 個(54%)** 可透過結構化欄位直達。
- 剩 28 個 query 無對應欄位映射(真開放語意,如「想要有家的感覺」)→ B/C 戰場。
- → A 不是小修補:**零後端、零重訓直接解掉一半以上召回缺口**,且界線清楚。

### 路線 A:純 edge,混合檢索(結構化過濾 + 向量召回)— 不動模型

把 ~15 個結構化布林欄位 + 一張「隱喻→特徵」對照表接進召回:
query 解析出意圖特徵(怕熱→need balcony/window;不想買水→need water_dispenser)
→ 結構化預過濾候選 → 再對縮小後的候選跑 bi-encoder 向量排序。

- **可行性**:零後端、純前端、零重訓。現役 `semanticExpandQuery`(口語→特徵詞映射表)
  已存在於 `inference-worker.js`,擴充它即可。`filterHardExclusions` 已是結構化過濾骨架。
- **預期效果**:直接解根因 1。「不想出門買水」→ `water_dispenser=true` 的 3 間能 100% 召回。
- **天花板**:只對「能映射到明確特徵」的隱喻有效;真開放語意(「想要有家的感覺」)仍靠向量。
- **成本**:低(1–2 PR)。**風險最低、投報率最高的第一步。**

### 路線 B:edge 內換更好的小 encoder + 領域微調召回目標

不是換更大,是換**更適合語意檢索**的小 encoder(如 e5-small/bge-small 級,~100MB)
+ 用 query↔特徵正樣本**重訓召回目標**(對比學習,正樣本 = query 的 GT 房源)。
房源 text 改餵 `ce_text`(富化版,已含 `有陽台 對外窗`)。

- **可行性**:仍 edge(~100MB 在載入優化後可接受);需 Colab 重訓。
- **預期效果**:解根因 1+2,但受 edge 體積上限約束,天花板仍在。
- **成本**:中(改 text 來源 + 重訓 + 蒸餾 + 評估閉環,3–5 PR,需 Colab)。
- **風險**:重訓可能再撞塌縮(stage4 教訓);需嚴守 ab_eval 獨立 gate。

### 路線 C:加輕量後端 — 大 encoder / LLM 語意層(離 edge)

只把 query embedding(或 LLM 意圖解析)放到一個輕量 API,房源向量仍預算好。
或更激進:LLM 直接做 query→特徵抽取 + rerank。

- **可行性**:**放棄零後端**(roadmap 核心 constraint);需部署 + 成本 + 延遲。
- **預期效果**:語意理解上限最高(大 encoder 領域微調 / LLM 常識推理)。
- **成本**:高(後端服務 + 部署 + 持續成本)。**重大架構決策。**
- **關鍵問題**:demo 規模(974 房源、個人專題)真需要嗎?投報率存疑。

## 四、建議

**先做路線 A(零成本、解根因 1、風險最低),用它的結果重新量天花板**,
再決定要不要走 B 或 C。理由:

1. 根因 1(結構化訊號沒進召回)是**最便宜且最確定**的一塊,A 直接解。
2. A 做完後重跑診斷,若隱喻召回大幅改善 → 可能根本不需要離 edge。
3. 若 A 後仍有明確語意鴻溝(開放語意),那才是 B/C 的戰場,且屆時有 A 的 baseline
   能誠實衡量 B/C 的增量值不值得放棄零後端。

**反對直接跳 C**:實測已證「換大 encoder」非萬靈丹(e5 零樣本更差),
且 demo 規模下放棄零後端的代價(部署/成本/複雜度)未必換得到等值語意提升。

## 五、裁示結果

**已拍板:先 A → 重量天花板 → 再決 B/C。**

## ⚠️ 五點五、實作前查證翻轉(2026-06-28)— A 幾乎無價值,真瓶頸在評估集

開始實作 A 前,逐欄查證結構化欄位的「區辨力」(base rate),結論翻轉:

### 結構化欄位 base rate(True 率)

| 欄位 | True 率 | 區辨力 |
|---|---|---|
| `has_balcony` | **95.0%** | 無(幾乎全有) |
| `has_elevator` | 88.4% | 無 |
| `has_window` | 80.5% | 無 |
| `has_subsidy` | 68.3% | 弱 |
| `has_waste_disposal` | 67.8% | 弱 |
| `water_dispenser` | 18.4% | **有** |
| `has_parking` | 9.0% | **有** |
| `is_rooftop` / `is_wooden_partition` / `private_washer` | 0% | 壞欄位(全 False) |

→ 只有 **2 個欄位有區辨力**。能映射到有區辨力欄位的語意 query 只有 **3/78**,救 5 個 relevant。
→ **第四節「54% 可結構化救回」是假象**:has_balcony 95% True → 「任何 query 配 has_balcony」
   幾乎必中 = 假陽性。修正後 A 的真實覆蓋 ≈ 3/78 query,**A 不值得做。**

### 更深的根問題:GT 本身沒有客觀依據

查 `build_ab_eval_queries.py`:GT `relevant_idxs` 來源 = `recommendation_train.json`
的合成標註,經 fuzzy-join 對到 idx;且 query 入選條件是「rule-based recall **漏掉**
≥1 relevant」(刻意篩 rule-based 必敗 = selection bias)。

「怕悶熱→9 間」在**任何**結構化欄位上都跟其他 916 間有陽台的房源無法區分
→ 這 9 間「為何相關」的依據只在訓練標註裡,不在客觀房源特徵。
這正是記憶警告的「訓練資料 100% 模板合成 + 正樣本自我參照 + fuzzy-join 24.4% caveat」。

### 結論:真瓶頸是「沒有可信的 ground truth」

在一個 GT 來自合成自我參照 + selection bias 的評估集上,**A/B/C 任何路線的數字
升降都不代表真實語意理解進步**(stage4 的「同源假象」「越破越退」全是此根問題切面)。
→ **動任何模型/檢索架構前,真正該先做的是「建一個有客觀依據的評估集」。**
   否則必重蹈 stage4:花力氣重訓/改架構,然後被假數字騙或發現數字不動。

## ⚠️ 五點六、最根本結論(2026-06-28)— 資料同質性是語意理解的先天天花板

準備建評估集前,先量化「有多少口語意圖在這 974 房源上**存在可區辨的正確答案**」。
一個意圖只有在它對應的特徵能把房源池**有意義地切分**(非 60–95% 房源都有)才可評估:

| 口語意圖 | 對應特徵 | 命中率 | 可評估? |
|---|---|---|---|
| 不想出門買水 | 飲水機 | 18.4% | ✅ |
| 要停機車 | 車位 | 9.0% | ✅ |
| 想曬棉被 | 曬衣場 | 7.4% | ✅ |
| 報稅/補助 | 可報稅 | 42.4% | ✅ |
| 不想追垃圾車 | 垃圾代收 | 67.8% | ❌ 近乎全有 |
| 怕熱/悶 | 陽台 | **95.0%** | ❌ 近乎全有 |
| 怕悶 | 對外窗 | 80.5% | ❌ |
| 不想爬樓梯 | 電梯 | 88.4% | ❌ |
| 想開伙 | 可開伙 | 65.8% | ❌ |
| 想養寵物 | 可養 | 62.7% | ❌ |
| 怕吵 | 氣密窗 | 66.7% | ❌ |
| 獨立電錶 | 台電 | 0% | ❌ 壞欄位 |

→ **12 個常見口語意圖只有 4 個有可區辨答案。**

### 根本結論:這不是模型/架構/評估集問題,是資料同質性

興大周邊學生套房**高度同質**:95% 有陽台、88% 有電梯、80% 有窗、66% 可開伙。
所以大部分「口語需求」在這個房源池裡**沒有有意義的區分答案** —— 問「怕熱」,
95% 房源都有陽台,沒有哪間特別涼可推。

這一次性解釋了所有先前撞牆:
- **stage4 越破模板越退**:在沒有區辨答案的維度上硬教「語意」,只破壞有區辨維度的能力。
- **評估集只能合成自我參照**:真實資料裡這些 query 沒有客觀正確答案可標。
- **換大 encoder 救不了**:沒有信號可學。

→ **在 974 筆同質興大套房上,「真正的語意理解推薦」先天空間很小。** 能做的語意區辨
   只剩少數低 base-rate 特徵(飲水機/車位/曬衣場/預算/距離),而那些用**結構化過濾**
   就解決,不需要語意模型。

### 真正能突破的前提(二選一或並行)

1. **擴大房源異質性**:接更多元的平台/區域,讓特徵分布不再 95% 同值
   —— 有了區辨性,語意模型才有發揮空間(但 roadmap constraint 限定中興周邊)。
2. **改變產品定位**:承認在同質池上「語意理解」邊際效益低,把力氣放在
   **結構化精準過濾 + 距離/預算/CP 值排序**(這些才是學生選房的真實決策維度),
   語意層只當錦上添花(口語→結構化意圖的 NLU),不追「向量語意理解」。

## 六、路線 A 實作計劃(structured-first 混合召回)— ⚠️ 經五點五/五點六已判定不值得做,保留供記錄

### 現狀(查證,非推測)

- 召回核心 `inference.js:782-793`:`allowed = Set(filterHardExclusions(...))`
  → `cosineTopK` 取「在 allowed 內」的純 cosine 前 30。
- `constraint-parser.js` 已解析部分意圖:`requireBalcony`(:33)、`requireWaste`(:40)、
  `requireWaterDispenser`(:143)等,但**僅用於 hard exclusion 過濾**,沒參與「保證召回」。
- 精排前的 keyword scoring(`inference.js:535-592`,註解自稱 "Option A")已有
  意圖→布林欄位映射(電梯/垃圾/陽台/窗/車位),但它**只對已進候選池的房源加分**,
  救不回「向量召回根本沒撈到」的房源 = 診斷的 4/6 MISS 來源。

### 缺口

純 cosine top-30 會漏掉「結構化欄位明確命中 query 意圖、但向量分數低」的房源。
這正是「不想出門買水→water_dispenser」3 間全 MISS 的機制。

### 改法(最小侵入,零後端零重訓)

1. **擴充意圖→欄位映射**(`constraint-parser.js`):補診斷缺的映射 ——
   怕熱/悶熱→`has_balcony`+`has_window`、淺眠/怕吵→隔音相關欄位、
   現有 `requireWaterDispenser` 等確認接上。產出 query 的「結構化需求集」。
2. **召回階段 structured-first union**(`inference.js` recall 段):
   候選 = (結構化需求命中的房源)∪(cosine top-30),仍受 hard exclusion 過濾,
   union 後若超量則用 cosine 分數排序截斷到一個合理上限(如 40)交給 CE 精排。
   結構化命中者給保底名額,確保不被向量分數低淹掉。
3. **kill-switch**:比照 `VECTOR_RECALL_ENABLED`,加 `STRUCTURED_RECALL_ENABLED`,
   關掉即回到純向量召回(可 A/B、可回退)。

### 驗證(守評估鐵則)

- **獨立 gate:ab_eval_queries all R@30 必須 ≥ 0.26**(現役 baseline,退步不落地)。
- 語意桶 R@30 應明顯上升(預期那 50/78 可結構化救回的 query 召回改善)。
- 本機 venv 複跑 `eval_vector_vs_rulebased.py` + 重跑診斷腳本對照前後。
- 前端 preview 親驗「怕悶熱」「不想出門買水」實際召回到對的房源。
- 一個確定性 self-check(假 query + 假房源欄位 → union 召回含結構化命中者)。

### 完成後

重跑全量診斷,量化 A 後的新天花板與「仍需開放語意」的殘餘缺口,
據此誠實評估 B/C 值不值得。

## 七、實施結果(2026-06-28,#99–#102)

- **#99 隱藏含義評估集**:Precision@30 為主防 Recall 虛降(符合條件 >30 時 Recall@30 天花板被分母壓死);GT 客觀算 + 隨機基準對照。現役 bi-encoder 15 個隱藏含義 9 個顯著理解(≥1.5x)。
- **#100/#101 路線 B 變體(補 text + 重訓)實證失敗**:把特徵線索補進召回 `text` + 設施 pair 重訓。本機正式 harness 複核:補 text 0.241、重訓 0.202,皆 <現役 ab_eval 0.26;退步元兇=降採樣誤砍 50% semantic GT 正樣本。**補 text/重訓都守不住鐵則。**
- **#102 路線 A 落地(結構化 boost)**:口語設施隱喻走結構化過濾 union(`parseFacilityIntents`),bi-encoder 不動。6 設施標靶 P@30 →1.0(含 bi-encoder 死路的台電),ab_eval 零影響、零模型風險。
- **最終結論**:語意交 bi-encoder(不動)、結構交過濾,各取所長。資料同質性的先天天花板未變(換大 encoder 仍是唯一突破真語意理解的路,屬離 edge 重大決策);但「設施隱喻召回」這塊已用零風險的結構化 boost 解決。
